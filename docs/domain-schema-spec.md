# Domain Schema Spec

**Document Status**: Draft  
**Version**: v1  
**Last Updated**: 2026-03-10

This document defines the core domain contracts for TagIt v1. These schemas are the canonical interface between scheduler, policy, persistence, Curia, and clients.

The intent of this spec is to stabilize:

* artifact structure
* validation boundaries
* persistence identity
* replay semantics
* execution eligibility

This is not yet a machine-readable JSON Schema bundle. It is the normative source for generating those schemas.

## 1. Design Rules

The following rules apply to all domain objects:

* all top-level objects must be serializable to JSON
* all timestamps use RFC 3339 / ISO 8601 in UTC
* all IDs are opaque strings generated by TagIt unless stated otherwise
* unknown fields must be rejected in strict validation mode
* every persisted object must declare `schema_version`
* objects used for execution decisions must be checksum-protected

## 2. Common Scalar Conventions

### 2.1 IDs

Recommended prefixes:

* `sess_` for session IDs
* `task_` for task IDs
* `art_` for artifact IDs
* `prop_` for proposal IDs
* `ballot_` for ballot IDs
* `dp_` for decision pack IDs
* `plan_` for execution plan IDs

### 2.2 Confidence

Allowed values:

* `high`
* `medium`
* `low`

### 2.3 Roles

Allowed producer roles:

* `executor`
* `planner`
* `reviewer`
* `senator`
* `arbitrator`
* `system`
* `human`

### 2.4 Strategy

Allowed task strategies:

These are internal scheduler strategy labels for task-graph nodes, not the user-facing `tagit run --mode` values. The current CLI modes are `rage`, `collab`, and `senate`.

* `direct`
* `relay`
* `curia`

### 2.5 Apply Mode

Allowed execution apply modes:

* `patch`
* `git_commit`
* `command_sequence`
* `manual`

`command_sequence` must be policy-gated and is not implicitly allowed in v1.

## 3. ArtifactEnvelope

`ArtifactEnvelope` is the required wrapper for every structured output produced or accepted by TagIt.

### 3.1 Schema

```json
{
  "id": "art_xxx",
  "kind": "proposal|ballot|debate_log|decision_pack|execution_plan|report",
  "schema_version": "v1",
  "producer": {
    "agent_id": "gemini-cli",
    "role": "senator",
    "run_id": "run_xxx"
  },
  "session_id": "sess_xxx",
  "task_id": "task_xxx",
  "created_at": "2026-03-10T10:00:00Z",
  "payload_schema": "tagit/proposal/v1",
  "payload": {},
  "attachments": [],
  "checksum": "sha256:..."
}
```

### 3.2 Fields

* `id`: required, unique artifact ID
* `kind`: required, logical artifact category
* `schema_version`: required, envelope version
* `producer`: required, producer identity metadata
* `session_id`: required
* `task_id`: required
* `created_at`: required
* `payload_schema`: required, versioned logical payload schema name
* `payload`: required, validated according to `payload_schema`
* `attachments`: optional, defaults to empty list
* `checksum`: required, checksum of canonicalized envelope content excluding `checksum`

### 3.3 Validation Rules

* `kind` and `payload_schema` must be compatible
* `session_id` and `task_id` must point to an existing logical context
* `producer.role` must be permitted for the artifact kind
* `checksum` mismatch makes the artifact invalid for execution use
* invalid attachments do not invalidate the envelope only if attachment policy marks them non-critical

## 4. AttachmentRef

Attachments are external blobs referenced by artifacts.

### 4.1 Schema

```json
{
  "name": "proposal.patch",
  "media_type": "text/x-diff",
  "blob_ref": "blobs/sess_xxx/task_xxx/art_xxx/proposal.patch",
  "size_bytes": 1823,
  "checksum": "sha256:...",
  "purpose": "diff|patch|snapshot|transcript|evidence"
}
```

### 4.2 Rules

* `blob_ref` must be stable and resolvable by Event Store
* attachment checksums are independent from envelope checksum
* attachments are optional and never replace required payload fields

## 5. AgentProfile

`AgentProfile` describes how TagIt launches and reasons about an external agent runtime.

### 5.1 Schema

```json
{
  "id": "claude-code",
  "display_name": "Claude Code",
  "command": "claude",
  "args": ["code"],
  "env_template": {
    "TAGIT_SESSION_ID": "{{session_id}}",
    "TAGIT_TASK_ID": "{{task_id}}"
  },
  "supports_mcp": true,
  "supports_json_output": false,
  "version_check": {
    "command": ["claude", "--version"]
  },
  "healthcheck": {
    "type": "version_check"
  },
  "capabilities": ["interactive", "patch_generation", "tool_use"],
  "timeout_defaults": {
    "startup_seconds": 20,
    "idle_seconds": 180,
    "max_runtime_seconds": 3600
  }
}
```

### 5.2 Rules

* `id`, `command`, and `display_name` are required
* `command` must resolve in daemon runtime environment
* `env_template` interpolation must be deterministic
* declared capabilities are advisory, not trust boundaries

## 6. TaskNodeSpec

`TaskNodeSpec` is the scheduler-facing static contract for a node in a task graph.

### 6.1 Schema

```json
{
  "id": "task_codegen",
  "title": "Generate repository implementation",
  "strategy": "relay",
  "inputs": [
    {
      "name": "design_doc",
      "source": "artifact",
      "required": true,
      "accepted_kinds": ["report"]
    }
  ],
  "expected_outputs": [
    {
      "kind": "report",
      "payload_schema": "tagit/report/v1"
    }
  ],
  "validation_rules": [
    "must_produce_artifact",
    "must_not_touch_forbidden_paths"
  ],
  "policy_hints": {
    "sensitive_paths": ["internal/auth/**"],
    "approval_required": false
  },
  "timeout": {
    "max_runtime_seconds": 1200
  },
  "retry_policy": {
    "max_attempts": 2,
    "backoff_seconds": 5
  },
  "dependencies": ["task_design"]
}
```

### 6.2 Rules

* `id`, `title`, and `strategy` are required
* `dependencies` must form an acyclic graph
* `expected_outputs` must be statically known
* `validation_rules` are scheduler-enforced, not advisory

## 7. Proposal

`Proposal` is the senator output in Curia scatter phase.

### 7.1 Schema

```json
{
  "proposal_id": "prop_xxx",
  "summary": "Introduce a daemon-backed PTY lifecycle manager.",
  "approach": "Use a session-scoped supervisor with event persistence.",
  "affected_files": [
    {
      "path": "internal/pty/manager.go",
      "change_type": "create"
    }
  ],
  "design_risks": [
    "PTY reconnect semantics differ on Windows ConPTY."
  ],
  "tradeoffs": [
    "Simpler supervisor logic but more event-store coupling."
  ],
  "estimated_steps": 6,
  "patch_plan": [
    "Add PTY manager interface.",
    "Persist process lifecycle events.",
    "Integrate scheduler callbacks."
  ],
  "confidence": "medium"
}
```

### 7.2 Rules

* `proposal_id` is required and unique within a task
* `affected_files` must not be empty for executable proposals
* `estimated_steps` must be a positive integer
* `confidence` uses common confidence enum

## 8. Ballot

`Ballot` is the anonymous review artifact for a proposal.

### 8.1 Schema

```json
{
  "ballot_id": "ballot_xxx",
  "target_proposal_id": "prop_xxx",
  "scores": {
    "correctness": 4,
    "safety": 5,
    "maintainability": 4,
    "scope_control": 3,
    "testability": 4
  },
  "critique": [
    "Recovery path for daemon restart is underspecified."
  ],
  "veto": false,
  "veto_reason": "",
  "confidence": "high"
}
```

### 8.2 Rules

* all score fields are required
* each score must be an integer from 1 to 5
* `veto_reason` is required when `veto` is `true`
* a ballot cannot target its own original proposal unless explicitly allowed by policy

## 9. DebateLog

`DebateLog` is the complete Curia case file used for audit and arbitration.

### 9.1 Schema

```json
{
  "debate_id": "debate_xxx",
  "session_id": "sess_xxx",
  "task_id": "task_xxx",
  "proposal_ids": ["prop_a", "prop_b", "prop_c"],
  "ballot_ids": ["ballot_1", "ballot_2", "ballot_3"],
  "dispute_summary": {
    "critical_veto": false,
    "close_score_gap": true,
    "key_conflicts": [
      "Whether execution should allow shell-based merge steps."
    ]
  },
  "quorum_reached_at": "2026-03-10T10:08:00Z",
  "arbitration_required": true
}
```

### 9.2 Rules

* `proposal_ids` must meet quorum threshold
* `ballot_ids` must only reference ballots for included proposals
* `arbitration_required` must be derivable from dispute rules

## 10. DecisionPack

`DecisionPack` is the arbitrator or human-arbitration result.

### 10.1 Schema

```json
{
  "decision_pack_id": "dp_xxx",
  "winning_mode": "merge",
  "selected_proposal_ids": ["prop_a", "prop_c"],
  "merged_rationale": [
    "Proposal A has the safer state model.",
    "Proposal C has the better workspace reintegration strategy."
  ],
  "rejected_reasons": [
    {
      "proposal_id": "prop_b",
      "reason": "Uses unrestricted command execution."
    }
  ],
  "execution_plan_id": "plan_xxx",
  "constraints": [
    "Do not modify paths outside internal/pty and internal/store.",
    "Human approval required before merge."
  ],
  "approval_required": true
}
```

### 10.2 Rules

* `winning_mode` must be one of `accept`, `merge`, `replace`
* `selected_proposal_ids` must be non-empty unless explicitly terminal
* `execution_plan_id` must resolve to a valid `ExecutionPlan`
* `approval_required` may be elevated by policy even if arbitrator says `false`

## 11. ExecutionPlan

`ExecutionPlan` is the only artifact eligible to authorize execution.

### 11.1 Schema

```json
{
  "plan_id": "plan_xxx",
  "goal": "Implement daemon-owned PTY lifecycle management with replay support.",
  "steps": [
    {
      "step_id": "step_1",
      "type": "file_edit",
      "description": "Add PTY manager interface and implementation.",
      "target_paths": ["internal/pty/manager.go"],
      "guardrails": [
        "Do not modify files outside declared paths."
      ]
    },
    {
      "step_id": "step_2",
      "type": "validation",
      "description": "Run unit tests for PTY and store packages.",
      "commands": ["go test ./internal/pty ./internal/store"]
    }
  ],
  "expected_files": [
    "internal/pty/manager.go",
    "internal/pty/manager_test.go"
  ],
  "forbidden_paths": [
    ".github/**",
    "infra/**"
  ],
  "required_checks": [
    "unit_tests",
    "artifact_validation"
  ],
  "apply_mode": "patch",
  "rollback_hint": "Discard worktree branch tagit-task-<id> if validation fails.",
  "human_approval_required": true
}
```

### 11.2 Step Types

Allowed `steps[].type` values:

* `file_edit`
* `command`
* `validation`
* `manual_gate`

### 11.3 Rules

* `plan_id`, `goal`, `steps`, and `apply_mode` are required
* every step must declare a deterministic intent
* `expected_files` must be a superset of writable target files
* `forbidden_paths` must be checked both pre-flight and during runtime
* `command` steps require policy approval
* if `human_approval_required` is `true`, scheduler must suspend before execution

## 15. RemoteCommand

`RemoteCommand` is the intent envelope sent from a remote endpoint back to TagIt.

### 15.1 Schema

```json
{
  "command_id": "rcmd_xxx",
  "source_endpoint_id": "gw_telegram_primary",
  "actor": "user:leo",
  "session_id": "sess_xxx",
  "task_id": "task_xxx",
  "action": "approve",
  "args": {},
  "issued_at": "2026-03-10T12:00:00Z",
  "signature": "base64:..."
}
```

Rules:

* remote commands are intent-level only
* `action` must be one of `approve`, `reject`, `pause`, `resume`, `cancel`, `retry`
* command acceptance requires authentication and policy revalidation

## 17. Validation Levels

TagIt should implement three validation levels:

### 13.1 Envelope Validation

Checks:

* required envelope fields
* checksum format
* known payload schema
* basic producer metadata

### 13.2 Payload Validation

Checks:

* strict field validation
* enum validation
* required field presence
* semantic field coupling

### 13.3 Policy Validation

Checks:

* allowed producer role for artifact kind
* allowed file/path scopes
* command-step restrictions
* approval requirements

Only artifacts that pass all required levels may influence execution.

## 18. Persistence Mapping

Recommended storage split:

* SQLite stores artifact index rows, IDs, kinds, session/task linkage, checksum, creation time, and validation status
* blob store keeps canonical JSON payloads and binary attachments

Recommended artifact index fields:

* `artifact_id`
* `kind`
* `payload_schema`
* `session_id`
* `task_id`
* `producer_agent_id`
* `producer_role`
* `created_at`
* `checksum`
* `validation_status`
* `blob_ref`

Recommended gateway index fields:

* `endpoint_id`
* `type`
* `enabled`
* `target`
* `delivery_status`
* `last_delivery_at`
* `last_error_code`

Recommended remote command audit fields:

* `command_id`
* `source_endpoint_id`
* `actor`
* `action`
* `session_id`
* `task_id`
* `issued_at`
* `accepted`
* `rejection_reason`

## 19. Compatibility Rules

### 15.1 Envelope Compatibility

`schema_version` changes only when the envelope shape changes.

### 15.2 Payload Compatibility

`payload_schema` changes when payload semantics or required fields change.

### 15.3 Forward Evolution

TagIt v1 should reject unknown required semantics by default rather than silently accepting them.

## 20. Open Items

The following remain intentionally unresolved and should become machine-readable schema tasks:

1. canonical JSON serialization rules for checksum generation
2. wildcard path pattern grammar for `forbidden_paths` and `expected_files`
3. whether score ranges remain `1..5` or become weighted decimals
4. whether `command` steps in `ExecutionPlan` are allowed in base policy
5. exact schema for `inputs` and `expected_outputs` in `TaskNodeSpec`
6. remote endpoint capability matrix by channel type
7. notification severity taxonomy and delivery escalation rules
